Adapter Flywheel

The order of operations for Nemotron Tinker.
1Data: upload SFT rows, rollout prompts, rewards, and eval sets.
2SFT: create LoRAs and teach each one a different target behavior.
3RL: collect rollouts, score them, and improve the same resident adapter when using the API RL path.
4Inference: sample, evaluate, add failures back to data, and repeat.
Use case view Gym-style sampling collects rollouts from the loaded adapter. Resident RL LoRA scores those rollouts and trains the same adapter on the same loaded base model.

Resident RL Goal

Teach each adapter to prefer a target answer, then sample the same adapter again.
1Goal: choose the exact behavior each adapter should prefer.
2Attempt: sample both resident adapters on the same prompts used for SFT.
3Reward: score each adapter's completion against its target text.
4Train together: submit one mixed RL job, then sample both again.
What this is showing SFT teaches the first answer for each prompt. RL reuses those exact prompts and pushes each adapter toward a new preferred answer.

Train Both Toward Goals

Use the same prompts as SFT, then train a new preference answer in one mixed resident RL job.
How to read this The prompt is reused from SFT. Before RL should look like the SFT answer; after RL should move toward the new target answer.
