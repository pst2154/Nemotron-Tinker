Adapter Flywheel

The order of operations for Nemotron Tinker.
1Data: upload SFT rows, rollout prompts, rewards, and eval sets.
2SFT: create a LoRA and teach the first behavior with train_steps.
3RL: collect rollouts, score them, and improve the same resident adapter.
4Inference: sample, evaluate, add failures back to data, and repeat.
Use case view Gym is a rollout collector. Resident RL LoRA is the API path that trains an adapter from rewards. NeMo-RL is for launching a separate dedicated RL job. They all belong to the RL/evaluation part of the flywheel.

RL + Evaluation Toolkit

One use case: collect rollouts, score them, then improve or launch RL.
1Gym / API: collect policy rollouts with token logprobs.
2Reward: score completions with rules, verifiers, or evals.
3Resident RL LoRA: train this service's adapter with RL loss inputs.
4NeMo-RL: launch a dedicated GRPO job when the run should own the GPUs.
Which path do I use? Use resident RL LoRA for the Tinker flywheel on small or mid-size adapters. Use Gym/API to collect rollout data. Use NeMo-RL when you want a separate dedicated RL training job.
